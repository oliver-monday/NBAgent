## PERSISTENCE GAPS

### GAP-1: `parlays.json` not committed by auditor.yml — HIGH

| Field | Value |
|-------|-------|
| File | `data/parlays.json` |
| Writer agent | `auditor.py` (`save()` — grades parlays in-place: writes `result`, `legs_hit`, `leg_results`) |
| Writer workflow | `auditor.yml` |
| Reader agents | `parlay.py` (reads historical parlays), `build_site.py` (renders Parlays tab with graded results) |
| Reader workflows | `analyst.yml` (parlay.py), `injuries.yml` (build_site.py) |
| Committed? | **NO** — auditor.yml commit step lists `picks.json`, `audit_log.json`, `audit_summary.json`, `audit_reports/`, `skipped_picks.json`, `skipped_picks_archive.json` but NOT `parlays.json` |

**Impact:** Parlay grading results are lost after every auditor run. The frontend Parlays tab never shows graded results (`result`, `legs_hit`, `leg_results` are always null on the committed copy). The `parlay.py` historical stats banner (parlay hit rate) has no data to display. This has been silently broken since the auditor parlay grading was implemented.

**Fix:** Add `"data/parlays.json"` to the auditor.yml commit step for loop.

## TODAY-CONDITION BUGS

### BUG-1: `Without [X]` line shown without verifying teammate is OUT today — HIGH

| Field | Value |
|-------|-------|
| Location | `agents/analyst.py`, `build_quant_context()`, lines 1049–1073 |
| Signal | `key_teammate_absent` → `Without [Teammate]` quant context line |
| Missing condition | No check that the named teammate is confirmed OUT/OFS in today's `injuries_today.json` |

**Current code (line 1051):**
```python
if kta and kta.get("n_games", 0) >= 3:
    tm_name = kta["teammate_name"]
    # ... renders Without [X] line — no injury status check
```

**Problem:** The `Without [X]` line is rendered whenever `key_teammate_absent` exists with ≥3 historical absence games, regardless of whether the teammate is playing today. If Reaves missed 7 games last month but is healthy today, the line `"Without Reaves (n=7): PTS T20=85%"` is still injected into Luka's quant context block. The Pick prompt rules (Gate 1: CONFIRMED-OUT REQUIREMENT) tell Claude to verify this — but the system is presenting the data as if it's contextually relevant when it may not be.

**Note:** The trend override was fixed (gated on `players_out_today` in `build_player_stats()`), but the `Without [X]` context line injection in `build_quant_context()` was not given the same gate. The data itself is correct — but showing it when the teammate is healthy is misleading context injection.

**Fix:** Pass `players_out_today` (or equivalent) to `build_quant_context()` and only render the `kta_line` when `teammate_name` is in the out set.
